2.1 Pourquoi et quand utiliser l'architecture hexagonale ?

L'architecture hexagonale n'est pas une solution universelle. Son adoption doit etre guidee par la nature du probleme, la complexite du domaine metier et la strategie de l'entreprise.

2.1.1 Domaine complexe

L'hexagonale prend tout son sens lorsque les regles metier sont complexes et constituent le coeur de la proposition de valeur.

Indicateurs qu'un domaine est complexe :

  • Les regles metier changent frequemment selon le contexte
  • Il existe de nombreuses conditions, exceptions et cas limites
  • Le vocabulaire metier est riche et specifique (ubiquitous language)
  • Les interactions entre entites metier sont non triviales
  • La logique de calcul ou de decision est sophistiquee
DomaineExemples de complexite
FinanceCalcul de risques, conformite reglementaire, orchestration de paiements multi-devises
AssuranceTarification dynamique, gestion des sinistres, regles d'eligibilite
LogistiqueOptimisation de tournees, gestion des stocks multi-entrepots, tracking temps reel
SanteProtocoles de soins, interactions medicamenteuses, parcours patient
E-commercePromotions combinables, gestion des retours, pricing dynamique

2.1 Spectre de complexite

+-----------------------------------------------------------------------+ | SPECTRE DE COMPLEXITE | | | | CRUD simple Logique moderee Domaine riche | | | | | | | v v v | | [ Spring Data REST ] [ Layered classique ] [ Hexagonale / DDD ] | | | | Ex: gestion de Ex: blog avec Ex: systeme | | parametres workflow simple de trading | +-----------------------------------------------------------------------+
Regle d'or : Si votre application est essentiellement du CRUD avec peu de logique metier, l'architecture hexagonale ajoutera de la complexite accidentelle sans benefice reel. Un simple @RestController + @Service + @Repository Spring Boot suffit.

2.1.2 Competences strategiques (DDD)

+-------------------------------------------------------------------+ | +------------------+ +------------------+ +-----------+ | | | CORE DOMAIN | | SUPPORTING | | GENERIC | | | | | | SUBDOMAIN | | SUBDOMAIN | | | | Avantage | | | | | | | | concurrentiel | | Necessaire mais | | Commodity | | | | | | pas differentiant| | | | | | --> HEXAGONALE | | --> Simplifie | | --> COTS | | | | --> DDD riche | | --> DDD light | | --> SaaS | | | +------------------+ +------------------+ +-----------+ | +-------------------------------------------------------------------+
L'architecture hexagonale brille pour le Core Domain : c'est la que la complexite metier justifie l'investissement en conception. Pour les sous-domaines generiques, on utilisera des solutions du marche integrees via des adapteurs.

2.1.3 Paradigmes de programmation

L'architecture hexagonale a ete concue dans un contexte oriente objet, mais elle peut s'adapter a d'autres paradigmes.

POO - Ajustement naturel

  • Interfaces incarnent les ports
  • Polymorphisme branche les adapteurs
  • Encapsulation protege les invariants

FP - Adaptation possible

  • Ports = signatures de fonctions
  • Adapteurs = implementations de fonctions
  • Interfaces fonctionnelles Java
// POO : Les INTERFACES incarnent naturellement les ports
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
}

// Le POLYMORPHISME permet de brancher differents adapteurs
public class JpaOrderRepository implements OrderRepository { /* ... */ }
public class InMemoryOrderRepository implements OrderRepository { /* ... */ }
// FP : Port = interface fonctionnelle
@FunctionalInterface
public interface FindOrderById {
    Optional<Order> apply(OrderId id);
}

// Adapteur = implementation de la fonction
FindOrderById jpaFindOrder = id -> Optional.ofNullable(
    jpaRepository.findById(id.value())
).map(OrderMapper::toDomain);
Programmation procedurale : Inadaptee - pas d'interfaces, pas de polymorphisme, pas d'encapsulation, couplage direct aux appels. Impossible d'inverser les dependances.

2.1.4 Microservices et Bounded Context

Chaque microservice peut etre modelise comme son propre hexagone, et les Bounded Contexts du DDD s'alignent naturellement sur les frontieres hexagonales.

Bounded Context Bounded Context Bounded Context "Commande" "Paiement" "Livraison" +-------------------+ +-------------------+ +-------------------+ | +----------+ | | +----------+ | | +----------+ | | | DOMAIN | | | | DOMAIN | | | | DOMAIN | | | | Order | | | | Payment | | | | Delivery | | | +----+-----+ | | +----+-----+ | | +----+-----+ | | [Ports/Adapt.] | ---> | [Ports/Adapt.] | ---> | [Ports/Adapt.] | +-------------------+ ACL +-------------------+ ACL +-------------------+

Regles d'integration entre hexagones :

  1. Chaque hexagone a son propre modele de domaine : un Order dans "Commande" n'est PAS le meme objet que dans "Livraison"
  2. La communication passe par des ports sortants : REST, messaging (Kafka, RabbitMQ), gRPC
  3. Des couches anti-corruption (ACL) traduisent entre les modeles de chaque bounded context
  4. Aucun hexagone ne partage sa base de donnees : chaque microservice a sa propre persistence
// Port sortant vers le service de paiement
public interface PaymentPort {
    PaymentResult requestPayment(OrderId orderId, Money amount);
}

// Adapteur driven - appel REST vers le microservice Paiement
@Component
public class PaymentRestAdapter implements PaymentPort {
    @Override
    public PaymentResult requestPayment(OrderId orderId, Money amount) {
        // ACL : on traduit nos objets domaine en DTO externe
        PaymentRequestDTO dto = new PaymentRequestDTO(
            orderId.value(), amount.value(), amount.currency().code()
        );
        PaymentResponseDTO response = restClient.post()
            .uri("/api/payments").body(dto)
            .retrieve().body(PaymentResponseDTO.class);
        // ACL : on retraduit la reponse externe en objet domaine
        return new PaymentResult(
            PaymentStatus.from(response.status()), response.transactionId()
        );
    }
}

2.2 Le Domaine

Le domaine est le coeur de l'hexagone. Il contient toute la logique metier et ne depend d'aucune technologie externe. C'est la couche la plus stable et la plus precieuse de l'application.

+---------------------------------------------------------------+ | HEXAGONE | | +-------------------------------+ | | | DOMAINE | | | | +----------+ +-----------+ | | | | | Agregats | | Value | | | | | | Entites | | Objects | | | | | +----------+ +-----------+ | | | | +----------+ +-----------+ | | | | | Domain | | Domain | | | | | | Events | | Services | | | | | +----------+ +-----------+ | | | | +----------+ | | | | | Ports | | | | | | (interf) | | | | | +----------+ | | | +-------------------------------+ | +---------------------------------------------------------------+

Composants du Domaine

  • Agregats - Unite de coherence
  • Entites - Objets avec identite
  • Value Objects - Immutables, auto-validants
  • Domain Events - Faits passes
  • Domain Services - Logique inter-agregats
  • Ports - Interfaces/contrats

Ce qui n'est PAS dans le Domaine

  • Annotations Spring (@Service, @Component)
  • Annotations JPA (@Entity, @Table)
  • Dependances sur des frameworks
  • Acces direct aux bases de donnees
  • Serialisation JSON / HTTP

2.2.1 Agregats et Racines d'Agregat

Un agregat est un groupe d'objets du domaine traite comme une unite de coherence. La racine d'agregat est le seul point d'entree pour interagir avec l'agregat.

Regles fondamentales des agregats :

  1. Les references externes ne pointent que vers la racine
  2. La coherence transactionnelle est garantie a l'interieur de l'agregat
  3. Les modifications passent toujours par la racine
  4. Un agregat est charge et persiste en entier
Agregat "Order" +-------------------------------------------+ | +------------------+ | | | Order (ROOT) | <-- seul point | | | - orderId | d'entree | | | - status | externe | | | - customerId | | | +--------+---------+ | | | contient | | +------------------+ | | | OrderLine | | | | - productRef | (reference, pas | | | - quantity | l'agregat | | | - unitPrice | Product entier) | | +--------+---------+ | | | utilise | | +------------------+ | | | Money (VO) | | | | - amount | | | | - currency | | | +------------------+ | +-------------------------------------------+

2.2.1 Agregat Order - Implementation

public class Order {
    private final OrderId id;
    private final CustomerId customerId;
    private OrderStatus status;
    private final List<OrderLine> lines;
    private final List<DomainEvent> domainEvents;

    // Constructeur prive : on passe par une factory method
    private Order(OrderId id, CustomerId customerId) {
        this.id = Objects.requireNonNull(id, "L'ID de commande est obligatoire");
        this.customerId = Objects.requireNonNull(customerId, "L'ID client est obligatoire");
        this.status = OrderStatus.DRAFT;
        this.lines = new ArrayList<>();
        this.domainEvents = new ArrayList<>();
    }

    // Factory method : point d'entree de creation
    public static Order create(CustomerId customerId) {
        OrderId id = OrderId.generate();
        Order order = new Order(id, customerId);
        order.registerEvent(new OrderCreated(id, customerId, Instant.now()));
        return order;
    }

    // Methode metier : ajout de ligne (invariant protege)
    public void addLine(ProductReference productRef, Quantity quantity, Money unitPrice) {
        assertDraft();
        lines.stream()
            .filter(line -> line.getProductRef().equals(productRef))
            .findFirst()
            .ifPresent(existing -> { throw new DuplicateProductException(productRef); });
        lines.add(new OrderLine(productRef, quantity, unitPrice));
    }

    // Methode metier : confirmation de commande
    public void confirm() {
        assertDraft();
        if (lines.isEmpty()) { throw new EmptyOrderException(id); }
        this.status = OrderStatus.CONFIRMED;
        registerEvent(new OrderConfirmed(id, calculateTotal(), Instant.now()));
    }

    private void assertDraft() {
        if (status != OrderStatus.DRAFT) {
            throw new OrderNotModifiableException(id, status);
        }
    }

    // Getters (pas de setters publics !)
    public OrderId getId() { return id; }
    public List<OrderLine> getLines() { return Collections.unmodifiableList(lines); }
}
Point cle : Remarquez que Order n'a aucun setter public. Toute modification passe par des methodes metier (addLine, confirm, cancel) qui protegent les invariants.

2.2.2 Entites

Une entite est un objet du domaine qui possede une identite unique et un cycle de vie. Son egalite se base sur son identite, pas sur ses attributs.

CaracteristiqueDescription
IdentiteUn identifiant unique et stable dans le temps
MutabiliteSon etat peut changer au cours de sa vie
Cycle de vieCreation, modification, eventuellement suppression
EgaliteDeux entites avec le meme ID sont la meme entite
// Identifiant type (Value Object pour l'ID)
public record OrderId(String value) {
    public OrderId {
        if (value == null || value.isBlank()) {
            throw new IllegalArgumentException("OrderId ne peut pas etre vide");
        }
    }
    public static OrderId generate() { return new OrderId(UUID.randomUUID().toString()); }
}

// Entite avec egalite sur l'identite
public class Customer {
    private final CustomerId id;
    private String name;
    private Email email;

    // Egalite basee sur l'identite uniquement
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Customer other)) return false;
        return id.equals(other.id);
    }
    @Override
    public int hashCode() { return id.hashCode(); }
}
Attention : Ne confondez pas entite DDD et entite JPA (@Entity). Une entite JPA est un concept de persistence. Une entite DDD est un concept metier. Dans l'architecture hexagonale, elles vivent dans des couches differentes.

2.2.3 Value Objects (VO)

Un Value Object est un objet du domaine sans identite propre, immutable, dont l'egalite se base sur tous ses attributs.

CaracteristiqueDescription
Pas d'identiteDeux VOs avec les memes valeurs sont interchangeables
ImmutableUne fois cree, il ne change jamais
Auto-validantIl garantit ses propres invariants a la construction
Egalite structurelleComparaison sur tous les champs
// VALUE OBJECT : Money (les Java Records sont parfaits)
public record Money(BigDecimal amount, Currency currency) {
    public static final Money ZERO_EUR = new Money(BigDecimal.ZERO, Currency.EUR);

    public Money {
        Objects.requireNonNull(amount, "Le montant est obligatoire");
        Objects.requireNonNull(currency, "La devise est obligatoire");
        if (amount.scale() > 2) {
            throw new IllegalArgumentException("Precision maximale : 2 decimales");
        }
    }

    // Operations (retournent un NOUVEAU VO a chaque fois)
    public Money add(Money other) {
        assertSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int factor) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(factor)), this.currency);
    }
}

// VALUE OBJECT : Email
public record Email(String value) {
    private static final Pattern EMAIL_PATTERN =
        Pattern.compile("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$");
    public Email {
        Objects.requireNonNull(value, "L'email est obligatoire");
        if (!EMAIL_PATTERN.matcher(value).matches()) {
            throw new InvalidEmailException(value);
        }
        value = value.toLowerCase().trim();
    }
}

2.2.4 Domain Events - Publication

// PORT SORTANT : Publication d'evenements
public interface DomainEventPublisher {
    void publish(DomainEvent event);
    void publishAll(List<DomainEvent> events);
}

// DANS LE USE CASE : on recupere et publie les evenements
@Service
public class ConfirmOrderUseCase {
    private final OrderRepository orderRepository;
    private final DomainEventPublisher eventPublisher;

    @Transactional
    public void execute(OrderId orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));
        order.confirm(); // La logique metier est dans le domaine
        orderRepository.save(order);
        eventPublisher.publishAll(order.pullDomainEvents());
    }
}

// ADAPTEUR : Publication via Spring Events
@Component
public class SpringDomainEventPublisher implements DomainEventPublisher {
    private final ApplicationEventPublisher springPublisher;
    @Override public void publish(DomainEvent event) { springPublisher.publishEvent(event); }
    @Override public void publishAll(List<DomainEvent> events) { events.forEach(this::publish); }
}

// LISTENER : Reaction a un evenement
@Component
public class SendConfirmationEmailOnOrderConfirmed {

2.2.5 Services de domaine

Un service de domaine contient de la logique metier qui n'appartient naturellement a aucune entite ni aucun VO. Il est sans etat (stateless).

SituationExemple
L'operation implique plusieurs agregatsTransfert de fonds entre deux comptes
La logique ne se rattache a aucune entiteCalcul de prix avec remises croisees
Un algorithme metier complexe et independantValidation de compatibilite de produits
// SERVICE DE DOMAINE : PricingService
public class PricingService {
    public Money calculateOrderTotal(
            List<OrderLine> lines, CustomerType customerType,
            DiscountPolicy discountPolicy) {
        Money subtotal = lines.stream()
            .map(OrderLine::lineTotal)
            .reduce(Money.ZERO_EUR, Money::add);
        BigDecimal customerDiscount = switch (customerType) {
            case PREMIUM -> new BigDecimal("0.10");   // 10%
            case BUSINESS -> new BigDecimal("0.05");  // 5%
            case STANDARD -> BigDecimal.ZERO;
        };
        Money afterDiscount = subtotal.multiply(
            BigDecimal.ONE.subtract(customerDiscount));
        return discountPolicy.apply(afterDiscount);
    }
}

// SERVICE DE DOMAINE : TransferService (2 agregats)
public class TransferService {
    public TransferResult transfer(Account source, Account destination, Money amount) {
        if (!source.hasSufficientFunds(amount))
            return TransferResult.insufficientFunds(source.getId(), amount);
        source.debit(amount);
        destination.credit(amount);
        return TransferResult.success(source.getId(), destination.getId(), amount);
    }
}
Distinction importante : Le service de domaine contient de la logique metier pure. Le service applicatif (use case) orchestre les appels et gere les transactions. Ne les confondez pas !

2.3 La couche Application

La couche Application orchestre les interactions entre le domaine et le monde exterieur. Elle contient les Use Cases (Application Services) qui definissent ce que l'application peut faire.

Responsabilites

  • Orchestration - Coordonne domaine et ports
  • Transactions - Delimite les frontieres transactionnelles
  • Securite - Verifie les autorisations
  • Conversion - Traduit commandes en appels domaine
  • Publication - Propage les evenements

Ce qui n'est PAS dedans

  • Logique metier (dans le domaine)
  • Acces direct a la BDD (via les ports)
  • Serialisation JSON, mapping HTTP (dans les adapteurs)
// Commande (DTO d'entree)
public record PlaceOrderCommand(
    String customerId, List<OrderLineCommand> lines
) {
    public record OrderLineCommand(
        String productReference, int quantity,
        double unitPrice, String currency
    ) {}
}

// Resultat (DTO de sortie)
public record PlaceOrderResult(String orderId, String status, String totalAmount) {}

2.3 Use Case - PlaceOrderUseCase

@Service
@Transactional
public class PlaceOrderUseCase {
    private final OrderRepository orderRepository;
    private final CustomerRepository customerRepository;
    private final PricingService pricingService;
    private final DomainEventPublisher eventPublisher;

    public PlaceOrderResult execute(PlaceOrderCommand command) {
        // 1. Recuperer les donnees necessaires via les ports
        CustomerId customerId = CustomerId.of(command.customerId());
        Customer customer = customerRepository.findById(customerId)
            .orElseThrow(() -> new CustomerNotFoundException(customerId));

        // 2. Executer la logique metier (deleguee au domaine)
        Order order = Order.create(customer.getId());
        for (var lineCmd : command.lines()) {
            order.addLine(
                ProductReference.of(lineCmd.productReference()),
                Quantity.of(lineCmd.quantity()),
                new Money(BigDecimal.valueOf(lineCmd.unitPrice()),
                          Currency.valueOf(lineCmd.currency()))
            );
        }
        order.confirm();

        // 3. Persister via le port de sortie
        orderRepository.save(order);

        // 4. Publier les evenements de domaine
        eventPublisher.publishAll(order.pullDomainEvents());

        // 5. Retourner le resultat
        Money total = order.calculateTotal();
        return new PlaceOrderResult(
            order.getId().value(), order.getStatus().name(),
            total.amount() + " " + total.currency()
        );
    }
}
Le Use Case orchestre sans contenir de logique metier. Chaque etape delegue au domaine ou passe par un port. C'est le chef d'orchestre, pas le musicien.

2.4 La couche Infrastructure

La couche Infrastructure contient les implementations concretes des ports secondaires. C'est la couche ou vivent les frameworks, les bases de donnees, les clients HTTP et toute la "plomberie" technique.

+-------------------------------------------------------------------+ | COUCHE INFRASTRUCTURE | | +-------------------+ +-------------------+ +---------------+ | | | JPA Repositories | | REST Clients | | Kafka | | | | (persistence) | | (appels externes) | | (messaging) | | | +-------------------+ +-------------------+ +---------------+ | | +-------------------+ +-------------------+ +---------------+ | | | Configuration | | Security | | Mapping | | | | Spring (beans) | | (JWT, OAuth) | | (Entite JPA) | | | +-------------------+ +-------------------+ +---------------+ | +-------------------------------------------------------------------+

Principes cles :

  1. L'infrastructure implemente les interfaces definies dans le domaine/application
  2. L'infrastructure ne contient pas de logique metier
  3. L'infrastructure est remplacable : on peut changer de BDD sans toucher au domaine
// Entite JPA (modele de persistence, PAS le domaine)
@Entity @Table(name = "orders")
public class OrderJpaEntity {
    @Id private String id;
    @Column(name = "customer_id", nullable = false) private String customerId;
    @Enumerated(EnumType.STRING) private String status;
    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true, fetch = FetchType.EAGER)
    @JoinColumn(name = "order_id")
    private List<OrderLineJpaEntity> lines = new ArrayList<>();
    protected OrderJpaEntity() {} // Requis par JPA
}

// Spring Data JPA Repository
public interface OrderJpaRepository extends JpaRepository<OrderJpaEntity, String> {
    List<OrderJpaEntity> findByCustomerId(String customerId);
}

2.4 Infrastructure - Mapper et Adapter

// MAPPER : Domaine <--> JPA
@Component
public class OrderMapper {
    public OrderJpaEntity toJpaEntity(Order order) {
        OrderJpaEntity entity = new OrderJpaEntity();
        entity.setId(order.getId().value());
        entity.setCustomerId(order.getCustomerId().value());
        entity.setStatus(order.getStatus().name());
        entity.setLines(order.getLines().stream()
            .map(this::toJpaEntity).toList());
        return entity;
    }

    public Order toDomain(OrderJpaEntity entity) {
        return Order.reconstitute(
            OrderId.of(entity.getId()),
            CustomerId.of(entity.getCustomerId()),
            OrderStatus.valueOf(entity.getStatus()),
            entity.getLines().stream().map(this::toDomain).toList(),
            entity.getCreatedAt()
        );
    }
}

// ADAPTEUR SECONDAIRE : Implemente le port du domaine
@Repository
public class JpaOrderRepositoryAdapter implements OrderRepository {
    private final OrderJpaRepository jpaRepository;
    private final OrderMapper mapper;

    @Override
    public Optional<Order> findById(OrderId id) {
        return jpaRepository.findById(id.value()).map(mapper::toDomain);
    }

    @Override
    public void save(Order order) {
        jpaRepository.save(mapper.toJpaEntity(order));
    }
}
Point important : Notez la separation entre Order (domaine) et OrderJpaEntity (persistence). L'entite JPA est un detail d'implementation. Le domaine ne sait meme pas que JPA existe.

2.5 Ports, Adapteurs et Anti-corruption Layer

Les ports et adapteurs sont le mecanisme central de l'architecture hexagonale. Ils permettent au domaine de communiquer avec le monde exterieur sans en dependre.

MONDE EXTERIEUR HEXAGONE MONDE EXTERIEUR (driving side) (driven side) +----------------+ +-------------------+ +--------------+ | REST Client | ---> | Port Primaire | | Base de | | (navigateur) | | (interface) | | donnees | +----------------+ | | | +--------------+ | Adapteur Primaire| | +----------------+ | (Controller) | +--------------+ | Message Queue | ---> | | | | Adapteur | | (Kafka) | | Application | | Secondaire | +----------------+ | Service ---->|----> | (JPA Repo) | | | | +--------------+ +----------------+ | DOMAINE | | | CLI | ---> | | | +--------------+ | (terminal) | | Port Secondaire | ---> | API externe | +----------------+ | (interface) | | (Stripe) | +-------------------+ +--------------+

Ports Primaires (IN)

Definissent ce que l'app SAIT FAIRE. Implementes par les Application Services.

Ports Secondaires (OUT)

Definissent ce dont l'app A BESOIN. Implementes par les adapteurs secondaires.

2.5 Ports Primaires et Secondaires

2.5.1 Ports Primaires (Driving)

// Ce que l'application sait faire
public interface OrderService {
    // Commandes (ecriture)
    PlaceOrderResult placeOrder(
        PlaceOrderCommand command);
    void confirmOrder(String orderId);
    void cancelOrder(String orderId,
        String reason);
    // Queries (lecture)
    OrderView getOrder(String orderId);
    List<OrderView> getOrdersByCustomer(
        String customerId);
}

2.5.2 Ports Secondaires (Driven)

// Ce dont l'application a besoin
public interface OrderRepository {
    Optional<Order> findById(OrderId id);
    void save(Order order);
}

public interface NotificationSender {
    void sendOrderConfirmation(
        Email to, OrderId orderId, Money total);
}

public interface PaymentGateway {
    PaymentResult charge(
        CustomerId customerId, Money amount);
}

public interface InventoryChecker {
    boolean isAvailable(
        ProductReference ref, Quantity qty);
}
Le port primaire est implemente par un Application Service. Les adapteurs primaires (controllers, CLI) appellent ce port sans connaitre l'implementation.

2.5.3 Adapteurs Primaires (Driving)

Les adapteurs primaires appellent les ports primaires. Ils traduisent les requetes externes en appels vers l'application.

// ADAPTEUR PRIMAIRE : REST Controller
@RestController @RequestMapping("/api/orders")
public class OrderRestController {
    private final OrderService orderService; // Depend du PORT, pas de l'impl

    @PostMapping
    public ResponseEntity<PlaceOrderResult> placeOrder(
            @Valid @RequestBody PlaceOrderRequest request) {
        PlaceOrderCommand command = new PlaceOrderCommand(request.customerId(),
            request.lines().stream()
                .map(l -> new PlaceOrderCommand.OrderLineCommand(
                    l.productReference(), l.quantity(), l.unitPrice(), l.currency()))
                .toList());
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(orderService.placeOrder(command));
    }
}

// ADAPTEUR PRIMAIRE : Kafka Event Listener
@Component
public class OrderKafkaListener {
    private final OrderService orderService;
    @KafkaListener(topics = "payment-completed", groupId = "order-service")
    public void onPaymentCompleted(PaymentCompletedEvent event) {
        orderService.confirmOrder(event.orderId());
    }
}

// ADAPTEUR PRIMAIRE : CLI
@ShellComponent
public class OrderCliAdapter {
    private final OrderService orderService;
    @ShellMethod("Annuler une commande")
    public String cancelOrder(String orderId, String reason) {
        orderService.cancelOrder(orderId, reason);
        return "Commande %s annulee.".formatted(orderId);
    }
}
Observation cle : Les trois adapteurs (REST, Kafka, CLI) appellent le meme port OrderService. Le domaine n'a aucune idee de la facon dont il est invoque !

2.5.5 Anti-Corruption Layer (ACL)

L'Anti-Corruption Layer protege votre domaine des modeles externes. Elle traduit entre le langage de votre domaine et le langage du systeme externe.

VOTRE DOMAINE ACL SYSTEME EXTERNE (votre langage) (traduction) (leur langage) +-------------+ +--------------+ +------------------+ | Order | ------> | OrderMapper | -------> | PurchaseOrder | | OrderLine | | ACL | | LineItem | | Money | <------ | | <------- | PriceAmount | +-------------+ +--------------+ +------------------+

Quand utiliser une ACL :

  • Integration avec un systeme legacy qui a son propre modele
  • Appel a une API externe dont vous ne controlez pas le modele
  • Communication entre bounded contexts avec des langages differents
  • Protection contre les changements du modele externe
@Component
public class LegacyPaymentACL implements PaymentGateway {
    @Override
    public PaymentResult charge(CustomerId customerId, Money amount) {
        // TRADUCTION : domaine --> modele legacy
        LegacyPaymentRequest req = new LegacyPaymentRequest();
        req.setAmountInCents(amount.toMinorUnits());
        req.setCurrencyCode(amount.currency().code());
        req.setBuyerReference(customerId.value());
        req.setTransactionType(1); // 1 = charge

        LegacyPaymentResponse resp = legacyClient.process(req);

        // TRADUCTION : modele legacy --> domaine
        return switch (resp.getStatusCode()) {
            case 0 -> PaymentResult.success(resp.getTxnId(),
                Money.fromMinorUnits(resp.getProcessedAmount(), amount.currency()));
            case 1 -> PaymentResult.declined(resp.getStatusMessage());
            default -> PaymentResult.failed("Erreur: " + resp.getStatusMessage());
        };
    }
}

2.6 Les dependances en hexagonale

La regle d'or : toutes les dependances pointent vers le domaine. Le domaine ne depend de rien d'exterieur.

Le Principe d'Inversion des Dependances (DIP)

SANS inversion (classique)

Controller | v Service | v Repository (impl) | v Base de donnees

AVEC inversion (hexagonale)

Controller (Adapteur Primaire) | v Port Primaire (interface) ^ | Application Service (impl) | v Port Secondaire (interface) ^ | Repository Adapter (impl) | v Base de donnees
DeVersType de dependance
Adapteur PrimairePort Primaire (interface)Appelle
Application ServicePort Primaire (interface)Implemente
Application ServicePort Secondaire (interface)Utilise
Adapteur SecondairePort Secondaire (interface)Implemente
DomaineRIENAucune dependance sortante

2.6 Structure de packages recommandee

com.example.order/ | +-- domain/ <-- Coeur metier (0 dependance externe) | +-- model/ <-- Objets du domaine | | +-- Order.java <-- Racine d'agregat | | +-- OrderLine.java, OrderId.java, Money.java, Quantity.java | +-- port/ | | +-- in/ <-- Ports Primaires (driving) | | | +-- OrderService.java | | +-- out/ <-- Ports Secondaires (driven) | | +-- OrderRepository.java, PaymentGateway.java, NotificationSender.java | +-- service/ <-- Services de domaine | +-- event/ <-- Domain Events | +-- exception/ <-- Exceptions metier | +-- application/ <-- Orchestration (depend du domaine seulement) | +-- command/ <-- PlaceOrderCommand, CancelOrderCommand | +-- query/ <-- GetOrderQuery, OrderView | +-- usecase/ <-- PlaceOrderUseCase, ConfirmOrderUseCase | +-- infrastructure/ <-- Plomberie technique +-- adapter/ | +-- in/rest/ <-- OrderRestController | +-- in/kafka/ <-- OrderKafkaListener | +-- in/cli/ <-- OrderCliAdapter | +-- out/persistence/ <-- JpaOrderRepositoryAdapter, OrderMapper | +-- out/payment/ <-- StripePaymentGateway, LegacyPaymentACL | +-- out/notification/ <-- SmtpNotificationSender | +-- out/messaging/ <-- SpringDomainEventPublisher +-- config/ <-- DomainConfiguration, PersistenceConfiguration
Regle d'or : Le package domain/ n'importe RIEN des packages application/ ou infrastructure/. Le flux de dependances est toujours : infrastructure -> application -> domain.

2.6 Validation automatique avec ArchUnit

On peut automatiser la verification des regles de dependances avec ArchUnit :

@AnalyzeClasses(packages = "com.example.order")
public class HexagonalArchitectureTest {

    // Le domaine ne depend de RIEN d'exterieur
    @ArchTest
    static final ArchRule domain_should_not_depend_on_application =
        noClasses().that().resideInAPackage("..domain..")
            .should().dependOnClassesThat().resideInAPackage("..application..");

    @ArchTest
    static final ArchRule domain_should_not_depend_on_infrastructure =
        noClasses().that().resideInAPackage("..domain..")
            .should().dependOnClassesThat().resideInAPackage("..infrastructure..");

    // Le domaine ne depend pas de Spring
    @ArchTest
    static final ArchRule domain_should_not_use_spring =
        noClasses().that().resideInAPackage("..domain..")
            .should().dependOnClassesThat().resideInAPackage("org.springframework..");

    // Le domaine ne depend pas de JPA
    @ArchTest
    static final ArchRule domain_should_not_use_jpa =
        noClasses().that().resideInAPackage("..domain..")
            .should().dependOnClassesThat().resideInAPackage("jakarta.persistence..");

    // L'application ne depend pas de l'infrastructure
    @ArchTest
    static final ArchRule application_should_not_depend_on_infrastructure =
        noClasses().that().resideInAPackage("..application..")
            .should().dependOnClassesThat().resideInAPackage("..infrastructure..");
}
ArchUnit s'execute comme un test JUnit classique. Si un developpeur ajoute un import interdit (ex: JPA dans le domaine), le build echoue automatiquement. C'est le gardien de vos regles architecturales.

Synthese du Module 2

1. Quand utiliser l'hexagonale ?

  • Domaine complexe (finance, sante, logistique)
  • Core Domain (pas les sous-domaines generiques)
  • POO naturellement, FP possible, procedurale inadaptee
  • Microservices = chaque service est un hexagone

2. Le Domaine contient

  • Agregats (racine = seul point d'entree)
  • Entites (identite, cycle de vie)
  • Value Objects (immutables, auto-validants)
  • Domain Events + Services de domaine

3. Couche Application

  • Use Cases / Application Services
  • Orchestre, ne contient PAS de logique metier
  • Gere les transactions
  • CQRS leger : Command + Query Handlers

4. Ports & Adapteurs

  • Primaires = ce que l'app SAIT FAIRE (in)
  • Secondaires = ce dont l'app A BESOIN (out)
  • ACL = traducteur vers le monde externe
  • Tout pointe vers le domaine (DIP)

5. Infrastructure

  • Implementations concretes des ports
  • JPA entities != Domain entities
  • Configuration Spring, Kafka, REST clients

6. Dependances

  • TOUT pointe vers le domaine
  • Le domaine depend de RIEN
  • DIP est la cle de voute
  • ArchUnit verifie automatiquement
