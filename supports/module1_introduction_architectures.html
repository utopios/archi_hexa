1.1 C'est quoi l'architecture logicielle ?

Une definition pragmatique

L'architecture logicielle, ce n'est pas un diagramme PowerPoint qu'on dessine en debut de projet et qu'on oublie ensuite. Ce n'est pas non plus le choix entre "monolithe" et "microservices".

L'architecture logicielle, c'est l'ensemble des decisions structurantes d'un systeme -- celles qui sont couteuses a changer une fois prises.
-- Ralph Johnson (co-auteur du GoF)

Ces decisions incluent :

  • Comment le code est organise (modules, packages, couches)
  • Comment les composants communiquent entre eux
  • Ou se trouvent les frontieres entre ce qui change souvent et ce qui est stable
  • Comment les dependances circulent dans le systeme

Architecture vs Design

ArchitectureDesign (conception)
PorteeStructure globale du systemeStructure locale d'un composant
Cout du changementEleveModere a faible
ExemplesChoix des couches, communication inter-modules, strategie de persistanceDesign patterns, nommage, structure d'une classe
Qui decideEquipe / architecteDeveloppeur
"Architecture is about the important stuff. Whatever that is." -- Martin Fowler

Pourquoi l'architecture compte

Equipe A (pas d'architecture) : chaque nouvelle fonctionnalite prend 3x plus de temps, bugs en cascade, personne n'ose toucher le code.

Equipe B (architecture pensee) : rythme constant, nouveaux devs productifs en quelques jours, tests fiables.

La difference tient a : Maintenabilite Testabilite Evolutivite

Idees recues a combattre

Idee recueRealite
"L'architecture, c'est le framework"Spring Boot, Quarkus, Micronaut sont des details d'implementation. L'architecture, c'est ce qui reste si on les retire.
"L'architecture, c'est l'infrastructure"Kubernetes, Docker, AWS ne sont pas l'architecture de votre application. Ce sont des choix de deploiement.
"On fait d'abord les features, on architecturera plus tard"Les decisions architecturales prises par defaut (ou par accident) sont les plus couteuses a corriger.
"L'architecture ralentit le developpement"Une bonne architecture accelere le developpement a moyen et long terme. Le cout initial est un investissement.
A retenir : Le framework est un detail. La base de donnees est un detail. Le domaine metier est la raison d'etre de l'application.

1.2 Au-dela du CRUD traditionnel

La structure que tout developpeur Java connait

src/main/java/com/example/
    controller/
        OrderController.java
    service/
        OrderService.java
    repository/
        OrderRepository.java
    model/
        Order.java
HTTP Request  → Controller → Service → Repository → Database
HTTP Response ← Controller ← Service ← Repository ← Database
Regardons un exemple concret : une application de gestion de commandes pour identifier les problemes de cette approche.

CRUD : L'entite et le repository

L'entite (model)

@Entity
@Table(name = "orders")
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String customerEmail;
    private String productName;
    private int quantity;
    private double unitPrice;
    private double totalPrice;
    private String status; // "PENDING", "CONFIRMED", "SHIPPED"

    @Temporal(TemporalType.TIMESTAMP)
    private Date createdAt;
}

Le repository

@Repository
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByCustomerEmail(String email);
    List<Order> findByStatus(String status);
}

CRUD : Le service

@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final JavaMailSender mailSender;
    private final RestTemplate restTemplate;

    @Transactional
    public Order createOrder(Order order) {
        // "Logique metier" melangee avec l'infrastructure
        order.setTotalPrice(order.getUnitPrice() * order.getQuantity());
        order.setStatus("PENDING");
        order.setCreatedAt(new Date());

        // Verification du stock via API externe
        StockResponse stock = restTemplate.getForObject(
            "http://inventory-service/api/stock/" + order.getProductName(),
            StockResponse.class
        );

        if (stock.getAvailable() < order.getQuantity()) {
            throw new RuntimeException("Stock insuffisant");
        }

        Order saved = orderRepository.save(order);

        // Envoi d'email de confirmation
        SimpleMailMessage message = new SimpleMailMessage();
        message.setTo(order.getCustomerEmail());
        message.setSubject("Commande confirmee #" + saved.getId());
        message.setText("Votre commande a ete enregistree.");
        mailSender.send(message);

        return saved;
    }
}
Probleme : Logique metier, appels HTTP, envoi d'emails, persistance -- tout est melange dans un seul service !

Les 4 problemes du CRUD

1. Domaine anemique

L'entite n'est qu'un sac de donnees -- des getters/setters, aucune logique. C'est l'anti-pattern Anemic Domain Model (Martin Fowler).

2. God Classes

Le service fait tout : calcul metier, appels HTTP, envoi d'emails, persistance. Apres un an : 2000 lignes. Personne ne veut y toucher.

3. Couplage fort

Pour tester la logique metier, il faut demarrer Spring, une BDD, un serveur SMTP, un service HTTP externe. On teste le cablage, pas la logique.

4. Impossible de changer

Remplacer JPA par jOOQ ? L'email par Slack ? Ajouter gRPC ? Il faut reecrire le service. La logique est soudee a l'infrastructure.

Il doit y avoir une meilleure facon de faire.

1.3 Architecture en couches DDD

Les 4 couches d'Eric Evans (2003)

┌─────────────────────────────────────────────┐
│           PRESENTATION (UI / API)           │
│  Controllers REST, CLI, pages web, GraphQL  │
├─────────────────────────────────────────────┤
│              APPLICATION                     │
│  Cas d'utilisation, orchestration, DTOs     │
│  Ne contient PAS de logique metier          │
├─────────────────────────────────────────────┤
│                DOMAINE                       │
│  Entites, Value Objects, Domain Services    │
│  Regles metier, invariants                  │
│  C'est le COEUR de l'application            │
├─────────────────────────────────────────────┤
│            INFRASTRUCTURE                    │
│  JPA, JDBC, SMTP, HTTP clients, fichiers   │
│  Implementation des interfaces du domaine   │
└─────────────────────────────────────────────┘

         │        │        │
         ▼        ▼        ▼
    Dependances de haut en bas uniquement
Regle fondamentale : Les dependances vont de haut en bas. Jamais de bas en haut.

DDD : L'entite riche

Couche Domaine -- l'entite avec sa logique

// domain/model/Order.java
public class Order {

    private Long id;
    private CustomerEmail customerEmail;  // Value Object
    private ProductName productName;      // Value Object
    private Quantity quantity;             // Value Object
    private Money unitPrice;              // Value Object
    private Money totalPrice;
    private OrderStatus status;
    private LocalDateTime createdAt;

    private Order() {}

    public static Order create(CustomerEmail email, ProductName product,
                               Quantity qty, Money price) {
        Order order = new Order();
        order.customerEmail = email;
        order.productName = product;
        order.quantity = qty;
        order.unitPrice = price;
        order.totalPrice = price.multiply(qty.value());
        order.status = OrderStatus.PENDING;
        order.createdAt = LocalDateTime.now();
        return order;
    }

    public void confirm() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException(
                "Seule une commande PENDING peut etre confirmee. Status actuel : " + status
            );
        }
        this.status = OrderStatus.CONFIRMED;
    }

    public void ship() {
        if (this.status != OrderStatus.CONFIRMED) {
            throw new IllegalStateException(
                "Seule une commande CONFIRMED peut etre expediee."
            );
        }
        this.status = OrderStatus.SHIPPED;
    }
    // Pas de setters publics !
}

DDD : Value Object et Repository

Couche Domaine -- le Value Object Money

// domain/model/Money.java
public record Money(BigDecimal amount, Currency currency) {

    public Money {
        if (amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("Le montant ne peut pas etre negatif");
        }
        Objects.requireNonNull(currency, "La devise est obligatoire");
    }

    public static Money euros(double value) {
        return new Money(BigDecimal.valueOf(value), Currency.getInstance("EUR"));
    }

    public Money multiply(int factor) {
        return new Money(amount.multiply(BigDecimal.valueOf(factor)), currency);
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Impossible d'additionner des devises differentes");
        }
        return new Money(this.amount.add(other.amount), this.currency);
    }
}

Couche Domaine -- interface du repository

// domain/repository/OrderRepository.java
public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(Long id);
    List<Order> findByCustomerEmail(CustomerEmail email);
}

DDD : Application et Infrastructure

Couche Application -- le cas d'utilisation

// application/CreateOrderUseCase.java
@Service
@RequiredArgsConstructor
public class CreateOrderUseCase {

    private final OrderRepository orderRepository;
    private final StockChecker stockChecker;
    private final NotificationSender notificationSender;

    @Transactional
    public OrderDto execute(CreateOrderCommand command) {
        Order order = Order.create(
            new CustomerEmail(command.email()),
            new ProductName(command.product()),
            new Quantity(command.quantity()),
            Money.euros(command.unitPrice())
        );

        if (!stockChecker.isAvailable(order.getProductName(), order.getQuantity())) {
            throw new InsufficientStockException(order.getProductName());
        }

        Order saved = orderRepository.save(order);
        notificationSender.sendOrderConfirmation(saved);
        return OrderDto.from(saved);
    }
}
L'entite protege ses invariants
La logique metier est testable sans infrastructure
Le service orchestre sans contenir de logique
Limitation : Le domaine depend encore de la couche infrastructure. Il faut un mecanisme qui force l'isolation du domaine.

1.4 Architecture Hexagonale (Ports & Adapters)

En 2005, Alistair Cockburn propose l'Architecture Hexagonale. L'hexagone met l'application au centre et tout le reste a l'exterieur.

"Permettre a une application d'etre pilotee indifferemment par des utilisateurs, des programmes, des tests automatises ou des scripts batch, et d'etre developpee et testee isolement de ses eventuels peripheriques et bases de donnees." -- Alistair Cockburn

Les concepts cles

Le Domaine

Du Java pur. Ni Spring, ni JPA, ni HTTP. Rien du monde exterieur.

Les Ports

Interfaces Java definies dans le domaine. Contrats de communication.

Les Adapters

Implementations concretes qui connectent les ports au monde reel.

Port Entrant (Driving)Port Sortant (Driven)
DirectionL'exterieur appelle l'applicationL'application appelle l'exterieur
Qui definit le contrat ?Le domaineLe domaine
Qui implemente ?Le domaine (use case)L'infrastructure (adapter)
ExemplesCreateOrderUseCaseOrderRepository, NotificationSender

L'inversion de dependance en action

Architecture classique (dependance directe)

Service ──depends──▶ JpaRepository ──depends──▶ JPA/Hibernate

Le service connait JPA. Pour tester, il faut JPA.

Architecture hexagonale (dependance inversee)

                    ┌──────────────────────────┐
                    │       DOMAINE            │
                    │                          │
JpaOrderRepository  │   OrderRepository        │  CreateOrderUseCase
(Infrastructure)   ─┤─▶ (Interface/Port)  ◀────├── (Use Case)
                    │                          │
                    └──────────────────────────┘

L'infrastructure depend du domaine. PAS l'inverse.
C'est le Dependency Inversion Principle (le D de SOLID).

Structure de packages

com.example.order/
├── domain/                          # ZERO dependance externe
│   ├── model/                       # Entites, Value Objects, Enums
│   ├── port/
│   │   ├── in/                      # Ports entrants (CreateOrderUseCase...)
│   │   └── out/                     # Ports sortants (OrderRepository...)
│   ├── service/                     # Implementation des ports entrants
│   └── exception/
├── adapter/                         # TOUTE l'infrastructure
│   ├── in/rest/                     # Controllers REST
│   └── out/persistence/             # JPA, SMTP, HTTP clients
└── config/                          # Configuration Spring (injection)

Hexagonale : Les Ports

Port entrant (interface)

// domain/port/in/CreateOrderUseCase.java
public interface CreateOrderUseCase {
    OrderResult execute(CreateOrderCommand command);
}

public record CreateOrderCommand(
    String customerEmail,
    String productName,
    int quantity,
    double unitPrice
) {}

public record OrderResult(
    Long id,
    String status,
    BigDecimal totalPrice
) {}

Ports sortants (interfaces)

// domain/port/out/OrderRepository.java
public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(Long id);
}

// domain/port/out/StockChecker.java
public interface StockChecker {
    boolean isAvailable(ProductName product, Quantity quantity);
}

// domain/port/out/NotificationSender.java
public interface NotificationSender {
    void sendOrderConfirmation(Order order);
}

Hexagonale : Le service domaine

AUCUN import Spring, JPA, HTTP, ou quoi que ce soit d'infrastructure 
// domain/service/CreateOrderService.java
public class CreateOrderService implements CreateOrderUseCase {

    private final OrderRepository orderRepository;
    private final StockChecker stockChecker;
    private final NotificationSender notificationSender;

    public CreateOrderService(OrderRepository orderRepository,
                              StockChecker stockChecker,
                              NotificationSender notificationSender) {
        this.orderRepository = orderRepository;
        this.stockChecker = stockChecker;
        this.notificationSender = notificationSender;
    }

    @Override
    public OrderResult execute(CreateOrderCommand command) {
        // 1. Creer l'entite domaine
        Order order = Order.create(
            new CustomerEmail(command.customerEmail()),
            new ProductName(command.productName()),
            new Quantity(command.quantity()),
            Money.euros(command.unitPrice())
        );

        // 2. Verifier le stock via le PORT (pas via RestTemplate !)
        if (!stockChecker.isAvailable(order.getProductName(), order.getQuantity())) {
            throw new InsufficientStockException(order.getProductName());
        }

        // 3. Persister via le PORT (pas via JPA !)
        Order saved = orderRepository.save(order);

        // 4. Notifier via le PORT (pas via SMTP !)
        notificationSender.sendOrderConfirmation(saved);

        // 5. Retourner le resultat
        return new OrderResult(
            saved.getId(), saved.getStatus().name(), saved.getTotalPrice().amount()
        );
    }
}

Hexagonale : Les Adapters

Adapter entrant (REST Controller)

// adapter/in/rest/OrderRestController.java
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderRestController {

    private final CreateOrderUseCase createOrderUseCase; // Interface, pas impl !

    @PostMapping
    public ResponseEntity<OrderResponse> createOrder(@RequestBody CreateOrderRequest request) {
        CreateOrderCommand command = new CreateOrderCommand(
            request.email(), request.product(), request.quantity(), request.unitPrice()
        );
        OrderResult result = createOrderUseCase.execute(command);
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(new OrderResponse(result.id(), result.status(), result.totalPrice()));
    }
}

Adapter sortant (Persistance JPA)

// adapter/out/persistence/JpaOrderRepository.java
@Repository
@RequiredArgsConstructor
public class JpaOrderRepository implements OrderRepository {

    private final SpringDataOrderRepository springDataRepo;
    private final OrderPersistenceMapper mapper;

    @Override
    public Order save(Order domainOrder) {
        OrderJpaEntity jpaEntity = mapper.toJpaEntity(domainOrder);
        OrderJpaEntity savedEntity = springDataRepo.save(jpaEntity);
        return mapper.toDomainEntity(savedEntity);
    }

    @Override
    public Optional<Order> findById(Long id) {
        return springDataRepo.findById(id).map(mapper::toDomainEntity);
    }
}

Hexagonale : Configuration et Test

Configuration Spring (le cablage)

// config/BeanConfiguration.java
@Configuration
public class BeanConfiguration {

    @Bean
    public CreateOrderUseCase createOrderUseCase(
            OrderRepository orderRepository,
            StockChecker stockChecker,
            NotificationSender notificationSender) {
        return new CreateOrderService(orderRepository, stockChecker, notificationSender);
    }
}

Le test : du Java pur, sans Spring !

class CreateOrderServiceTest {
    // Pas de @SpringBootTest ! Pas de @MockBean ! Pas de contexte Spring !

    private CreateOrderService service;

    @BeforeEach
    void setUp() {
        var orderRepository = new InMemoryOrderRepository();
        var stockChecker = (StockChecker) (product, qty) -> true;
        var notificationSender = (NotificationSender) order -> {};
        service = new CreateOrderService(orderRepository, stockChecker, notificationSender);
    }

    @Test
    void should_create_order_with_correct_total() {
        var command = new CreateOrderCommand("client@test.com", "Laptop", 3, 999.99);
        OrderResult result = service.execute(command);
        assertThat(result.totalPrice()).isEqualByComparingTo(new BigDecimal("2999.97"));
        assertThat(result.status()).isEqualTo("PENDING");
    }

    @Test
    void should_reject_order_when_stock_insufficient() {
        stockChecker = (product, qty) -> false;
        service = new CreateOrderService(orderRepository, stockChecker, notificationSender);
        var command = new CreateOrderCommand("client@test.com", "Laptop", 100, 999.99);
        assertThatThrownBy(() -> service.execute(command))
            .isInstanceOf(InsufficientStockException.class);
    }
}
Ce test s'execute en millisecondes, sans aucune infrastructure. C'est la puissance de l'architecture hexagonale.

1.5 Architecture en Oignon (Onion Architecture)

En 2008, Jeffrey Palermo propose l'Onion Architecture : meme idee que l'hexagonale, mais en cercles concentriques.

┌──────────────────────────────────────────────────────────────────┐
│                        INFRASTRUCTURE                            │
│   UI, Tests, Frameworks, DB, File System, External Services      │
│                                                                  │
│    ┌──────────────────────────────────────────────────────┐      │
│    │                APPLICATION SERVICES                  │      │
│    │        Use Cases, Workflow Orchestration             │      │
│    │                                                      │      │
│    │    ┌──────────────────────────────────────────┐      │      │
│    │    │           DOMAIN SERVICES                │      │      │
│    │    │     Logique metier multi-entites        │      │      │
│    │    │                                          │      │      │
│    │    │    ┌──────────────────────────────┐      │      │      │
│    │    │    │        DOMAIN MODEL          │      │      │      │
│    │    │    │   Entities, Value Objects    │      │      │      │
│    │    │    └──────────────────────────────┘      │      │      │
│    │    └──────────────────────────────────────────┘      │      │
│    └──────────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────────┘

◄──────────────────── Les dependances pointent ────────────────────►
                      TOUJOURS vers le centre

Difference avec l'hexagonale

HexagonaleOignon
Domaine = un seul blocDomaine = Model + Services
Ports (in/out)Interfaces dans Domain Services
Adapters (in/out)Infrastructure (couche externe)
En pratique, les deux architectures menent a un code tres similaire. L'oignon offre simplement une granularite plus fine dans l'organisation du domaine.

1.6 Clean Architecture

En 2012, Robert C. Martin (Uncle Bob) synthetise les idees de l'hexagonale, de l'oignon et d'autres architectures en un modele unifie.

┌──────────────────────────────────────────────────────────────────┐
│                  FRAMEWORKS & DRIVERS                            │
│              Web, DB, Devices, External APIs                     │
│                                                                  │
│    ┌──────────────────────────────────────────────────────┐      │
│    │              INTERFACE ADAPTERS                       │      │
│    │        Controllers, Gateways, Presenters            │      │
│    │                                                      │      │
│    │    ┌──────────────────────────────────────────┐      │      │
│    │    │             USE CASES                     │      │      │
│    │    │     Application Business Rules           │      │      │
│    │    │                                          │      │      │
│    │    │    ┌──────────────────────────────┐      │      │      │
│    │    │    │          ENTITIES             │      │      │      │
│    │    │    │  Enterprise Business Rules   │      │      │      │
│    │    │    └──────────────────────────────┘      │      │      │
│    │    └──────────────────────────────────────────┘      │      │
│    └──────────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────────┘

         ═══════▶  The Dependency Rule  ◀═══════
         Le code source pointe TOUJOURS vers l'interieur
The Dependency Rule : Rien dans un cercle interieur ne peut connaitre quoi que ce soit d'un cercle exterieur. Cela inclut : les noms de fonctions, les classes, les variables.

Clean Architecture : Le code

// ENTITIES (cercle interieur) - Aucune annotation Spring, JPA, Jackson...
public class Order {
    private OrderId id;
    private Money total;
    private OrderStatus status;

    public void confirm() {
        if (status != OrderStatus.PENDING) {
            throw new OrderAlreadyConfirmedException(id);
        }
        this.status = OrderStatus.CONFIRMED;
    }
}

// USE CASES - Depend uniquement des Entities et des interfaces de sortie
public class CreateOrderInteractor implements CreateOrderInputBoundary {

    private final OrderGateway orderGateway;
    private final CreateOrderOutputBoundary presenter;

    @Override
    public void execute(CreateOrderRequest request) {
        Order order = Order.create(/* ... */);
        orderGateway.save(order);
        presenter.present(new CreateOrderResponse(order.getId()));
    }
}

// INTERFACE ADAPTERS - Traduction HTTP <-> Use Case
@RestController
public class OrderController {
    private final CreateOrderInputBoundary useCase;

    @PostMapping("/orders")
    public ResponseEntity<?> create(@RequestBody OrderApiModel apiModel) {
        CreateOrderRequest request = new CreateOrderRequest(
            apiModel.getEmail(), apiModel.getProduct()
        );
        useCase.execute(request);
        return ResponseEntity.created(/* ... */).build();
    }
}

Input/Output Boundaries

Controller ──▶ InputBoundary ──▶ Use Case Interactor ──▶ OutputBoundary ──▶ Presenter
               (interface)         (implementation)       (interface)       (implementation)

Quelle architecture choisir ?

SituationRecommandation

Ce qu'elles partagent toutes

1. Isoler le domaine metier

Le domaine ne depend de RIEN d'exterieur. Pas d'import javax.persistence.*, pas d'import org.springframework.*

2. Inverser les dependances

L'infrastructure depend du domaine grace a des interfaces, pas l'inverse. C'est le DIP (SOLID).

3. Separer "quoi" et "comment"

Le domaine dit quoi faire. L'infrastructure dit comment le faire.

4. Testabilite sans infrastructure

La logique metier se teste en < 1ms, sans Spring, sans BDD, sans rien.

// CE TEST FONCTIONNE avec les 4 architectures
@Test
void une_commande_de_3_articles_a_10_euros_coute_30_euros() {
    Order order = Order.create(
        new CustomerEmail("test@test.com"),
        new ProductName("Widget"),
        new Quantity(3),
        Money.euros(10.0)
    );
    assertThat(order.getTotalPrice()).isEqualTo(Money.euros(30.0));
}
// Temps d'execution : < 1ms. Zero infrastructure.
