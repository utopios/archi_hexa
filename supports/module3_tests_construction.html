3.1 La pyramide des tests hexagonale

L'architecture hexagonale rend la stratégie de test naturelle et évidente. Chaque couche correspond à un type de test.

            /\
           /  \
          / IT  \          Tests d'Intégration
         /--------\        (Infrastructure)
        / BDD/Acc. \       Tests d'Acceptance
       /------------\      (Application)
      /   Unitaires   \    Tests Unitaires
     /___________________\  (Domaine)

Vitesse & Volume

  • Unitaires >> BDD >> Intégration
  • Les plus nombreux à la base
  • Ultra-rapides (pas de contexte)

Coût

  • Unitaires << BDD << Intégration
  • Les moins coûteux à la base
  • Intégration = plus complexe
L'inversion de dépendances permet : domaine sans mock, application avec mocks simples, infrastructure avec outils d'intégration.

3.1.1 Tests couche domaine : Tests Unitaires

La couche domaine ne contient aucune dépendance externe : pas de Spring, pas de JPA, pas de framework HTTP. C'est du Java pur.

Conséquences directes

  • Ultra-rapides pas de démarrage de contexte
  • Déterministes pas d'I/O, pas de BDD
  • Sans mocks pas de dépendances à simuler
  • Les plus nombreux base de la pyramide

Quoi tester ?

  • Value Objects : égalité, validation, opérations
  • Entités : transitions d'état, règles métier
  • Domain Services : logique transverse
Règle d'or : si vous avez besoin de mocks pour tester votre domaine, c'est que votre domaine a trop de dépendances. Refactorisez.

Tester un Value Object : Money

Les Value Objects sont immutables et définis par leurs attributs. Voici le record Money :

public record Money(BigDecimal amount, Currency currency) { public Money { Objects.requireNonNull(amount, "Le montant ne peut pas etre null"); Objects.requireNonNull(currency, "La devise ne peut pas etre null"); amount = amount.setScale(2, RoundingMode.HALF_UP); } public static Money of(double amount, String currencyCode) { return new Money(BigDecimal.valueOf(amount), Currency.getInstance(currencyCode)); } public Money add(Money other) { assertSameCurrency(other); return new Money(this.amount.add(other.amount), this.currency); } public Money subtract(Money other) { assertSameCurrency(other); if (this.amount.compareTo(other.amount) < 0) throw new IllegalArgumentException("Le montant ne peut pas etre negatif"); return new Money(this.amount.subtract(other.amount), this.currency); } }

Tests du Value Object Money

class MoneyTest { @Nested @DisplayName("Creation") class Creation { @Test @DisplayName("cree un Money avec montant et devise") void shouldCreateMoney() { Money money = Money.of(10.50, "EUR"); assertThat(money.amount()).isEqualByComparingTo("10.50"); assertThat(money.currency().getCurrencyCode()).isEqualTo("EUR"); } @Test @DisplayName("refuse un montant null") void shouldRejectNullAmount() { assertThatNullPointerException() .isThrownBy(() -> new Money(null, Currency.getInstance("EUR"))) .withMessage("Le montant ne peut pas etre null"); } } @Nested @DisplayName("Operations arithmetiques") class Arithmetic { @Test @DisplayName("additionne deux montants de meme devise") void shouldAddSameCurrency() { Money a = Money.of(10.00, "EUR"); Money b = Money.of(5.50, "EUR"); Money result = a.add(b); assertThat(result.amount()).isEqualByComparingTo("15.50"); } @Test @DisplayName("refuse l'addition de devises differentes") void shouldRejectDifferentCurrencies() { Money eur = Money.of(10.00, "EUR"); Money usd = Money.of(5.00, "USD"); assertThatIllegalArgumentException() .isThrownBy(() -> eur.add(usd)) .withMessageContaining("devises differentes"); } } }
Aucun mock, aucun framework autre que JUnit + AssertJ. @Nested structure les tests par responsabilité.

3.1.2 Tests couche application : BDD

La couche application orchestre les cas d'utilisation via des ports (interfaces).

Stratégie de test

  • Mocker les driven ports (repositories, services externes)
  • Écrire les scénarios en Gherkin (lisibles par les métiers)
  • Les step definitions appellent les services applicatifs

Ce que le BDD apporte

  • Scénarios lisibles par les non-développeurs
  • Spécification exécutable
  • Pont entre les métiers et les développeurs
  • Passage direct du Gherkin au code
Le BDD est le pont entre les experts métier et les développeurs. Les scénarios Gherkin constituent un langage commun.
<!-- Dépendances Maven --> <dependency> <groupId>io.cucumber</groupId> <artifactId>cucumber-java</artifactId> <version>7.15.0</version> <scope>test</scope> </dependency> <dependency> <groupId>io.cucumber</groupId> <artifactId>cucumber-junit-platform-engine</artifactId> <version>7.15.0</version> <scope>test</scope> </dependency>

3.1.4 Tests d'architecture avec ArchUnit

ArchUnit valide automatiquement que les règles de dépendances hexagonales sont respectées dans la CI/CD.

@AnalyzeClasses(packages = "com.example", importOptions = ImportOption.DoNotIncludeTests.class) class HexagonalArchitectureTest { // Règle 1 : le domaine ne dépend de RIEN d'extérieur @ArchTest static final ArchRule domain_not_depend_on_app = noClasses().that().resideInAPackage("..domain..") .should().dependOnClassesThat() .resideInAPackage("..application.."); @ArchTest static final ArchRule domain_not_depend_on_infra = noClasses().that().resideInAPackage("..domain..") .should().dependOnClassesThat() .resideInAPackage("..infrastructure.."); // Règle 2 : le domaine ne dépend d'aucun framework @ArchTest static final ArchRule domain_not_use_spring = noClasses().that().resideInAPackage("..domain..") .should().dependOnClassesThat() .resideInAPackage("org.springframework.."); // Règle 3 : les ports sont des interfaces @ArchTest static final ArchRule ports_are_interfaces = classes().that().resideInAPackage("..port..") .should().beInterfaces(); // Règle 4 : architecture en oignon complète @ArchTest static final ArchRule onion = onionArchitecture() .domainModels("..domain.model..") .domainServices("..domain.service..") .applicationServices("..application..") .adapter("web", "..infrastructure.web..") .adapter("persistence", "..infrastructure.persistence..") .adapter("messaging", "..infrastructure.messaging.."); }
En cas de violation, le message indique exactement quelle classe enfreint quelle règle.

3.2 Construction Itérative

3.2.1 Retarder les décisions d'architecture

Concevez vos ports d'abord, implémentez les adapteurs réels en dernier. Utilisez des adapteurs in-memory pendant le développement.

JourPhaseDétail
Jour 1-3Domaine + Ports + InMemoryAdaptersApplication fonctionnelle, testable, démontrable
Jour 4-5JpaAdapter (PostgreSQL)Persistence réelle
Jour 6KafkaAdapterMessaging réel
Jour 7RestAdapter + OpenAPIExposition REST
Avantage : pas besoin d'installer PostgreSQL, Kafka, etc. pour commencer. Les démos aux stakeholders sont possibles dès le jour 1.

Port et adapteur in-memory

Le port (interface)

// Port secondaire (driven) public interface OrderRepository { Order save(Order order); Optional<Order> findById(OrderId id); List<Order> findByCustomerId( CustomerId customerId); void deleteById(OrderId id); }

Adapteur in-memory

public class InMemoryOrderRepository implements OrderRepository { private final Map<OrderId, Order> store = new ConcurrentHashMap<>(); @Override public Order save(Order order) { store.put(order.getId(), order); return order; } @Override public Optional<Order> findById(OrderId id) { return Optional.ofNullable(store.get(id)); } // ... autres méthodes }
Le domaine et la couche application sont immédiatement fonctionnels. Le choix de la base de données peut être différé.

3.2.2 Double boucle BDD/TDD

La double boucle combine une boucle externe (BDD, comportement métier) et une boucle interne (TDD, implémentation).

  BOUCLE EXTERNE (BDD)                  BOUCLE INTERNE (TDD)
  =====================                 =====================
  1. Ecrire un scénario     --------->  2. Ecrire un test unitaire
     Cucumber (RED)                        du domaine (RED)
        |                                      |
        |                                      v
        |                                  3. Implémenter le code
        |                                     minimal (GREEN)
        |                                      |
        |                                      v
        |                                  4. Refactoriser (REFACTOR)
        |                                      |
        v                                      |
  5. Le scénario            <---------  Boucle interne terminée
     passe (GREEN)
        |
        v
  6. Refactoriser si nécessaire
  7. Ecrire le prochain scénario -> retour à 1

Exemple : Appliquer un code promo (BDD)

Étape 1 : Écrire le scénario BDD (boucle externe - RED)

Feature: Appliquer un code promotionnel En tant que client Je veux appliquer un code promo a ma commande Afin de bénéficier d'une réduction Scenario: Appliquer un code promo valide de 20% Given un code promo "SUMMER20" offrant 20% de réduction And une commande avec un montant de 100.00 EUR When j'applique le code promo "SUMMER20" Then le montant après réduction est de 80.00 EUR Scenario: Refuser un code promo expiré Given un code promo "OLD10" expiré depuis le "2024-01-01" And une commande avec un montant de 100.00 EUR When je tente d'appliquer le code promo "OLD10" Then je reçois une erreur "Code promo expiré" Scenario: Cumul impossible de deux codes promo Given un code promo "PROMO1" offrant 10% de réduction And un code promo "PROMO2" offrant 15% de réduction And une commande avec un montant de 100.00 EUR And le code promo "PROMO1" est déjà appliqué à la commande When je tente d'appliquer le code promo "PROMO2" Then je reçois une erreur "Un seul code promo par commande"
Ce scénario échoue (RED) car le concept de code promo n'existe pas encore.

Boucle interne TDD : Value Object PromoCode

Étape 2 : RED -> GREEN -> REFACTOR

Test unitaire (RED)

class PromoCodeTest { @Test void shouldCreatePromoCode() { PromoCode promo = PromoCode .percentage("SUMMER20", 20, LocalDate.of(2025, 12, 31)); assertThat(promo.code()) .isEqualTo("SUMMER20"); assertThat(promo.isExpired()).isFalse(); } @Test void shouldDetectExpiredCode() { PromoCode promo = PromoCode .percentage("OLD10", 10, LocalDate.of(2024, 1, 1)); assertThat(promo.isExpired()).isTrue(); } @Test void shouldCalculateDiscount() { PromoCode promo = PromoCode .percentage("SUMMER20", 20, LocalDate.of(2025, 12, 31)); Money discounted = promo .applyTo(Money.of(100.00, "EUR")); assertThat(discounted.amount()) .isEqualByComparingTo("80.00"); } }

Implémentation (GREEN)

public record PromoCode( String code, int discountPercent, LocalDate expirationDate, boolean used ) { public PromoCode { if (code == null || code.isBlank()) throw new IllegalArgumentException( "Le code ne peut pas etre vide"); if (discountPercent <= 0 || discountPercent > 100) throw new IllegalArgumentException( "Pourcentage entre 1 et 100"); } public static PromoCode percentage( String code, int percent, LocalDate expiration) { return new PromoCode(code, percent, expiration, false); } public boolean isExpired() { return LocalDate.now() .isAfter(expirationDate); } public Money applyTo(Money original) { if (isExpired()) throw new IllegalStateException("expire"); double factor = 1.0 - (discountPercent / 100.0); double discounted = original.amount() .doubleValue() * factor; return Money.of(discounted, original.currency() .getCurrencyCode()); } }

Port et service applicatif (code promo)

Ports

3.2.4 Couches d'anti-corruption (ACL)

Lorsque vous intégrez des systèmes externes, protégez votre domaine contre les concepts étrangers.

  Système externe              ACL              Votre hexagone
  +--------------+        +---------+        +----------+
  | CustomerDTO  |  --->  | Mapper  |  --->  | Customer |
  | (XML, champs |  --->  | Adapter |  --->  | (Value   |
  |  techniques) |  --->  |         |  --->  |  Objects)|
  +--------------+        +---------+        +----------+
@Component public class LegacyBillingAntiCorruptionLayer { private static final DateTimeFormatter LEGACY_DATE_FORMAT = DateTimeFormatter.ofPattern("yyyyMMdd"); public Invoice toDomain(LegacyInvoiceResponse legacy) { return new Invoice( new InvoiceId(legacy.getInv_nbr()), new CustomerId(legacy.getCust_ref()), Money.of(legacy.getTtl_amt(), mapCurrency(legacy.getCurr_cd())), Money.of(legacy.getTax_amt(), mapCurrency(legacy.getCurr_cd())), mapStatus(legacy.getInv_sts()), parseDate(legacy.getCrt_dt()), parseDate(legacy.getDue_dt()), mapLines(legacy.getLn_items(), legacy.getCurr_cd())); } private String mapCurrency(String legacyCurrency) { return switch (legacyCurrency) { case "E" -> "EUR"; case "D" -> "USD"; case "P" -> "GBP"; default -> legacyCurrency; }; } }
L'ACL est toujours dans la couche infrastructure, jamais dans le domaine. Le domaine ne connaît pas les concepts du système externe.

Résumé du Module 3

Pyramide des Tests

  • Domaine : zéro mock, Java pur, ultra-rapide
  • Application : BDD/Cucumber, driven ports mockés
  • Infrastructure : Testcontainers, vrais adapteurs
  • Architecture : ArchUnit, validation automatique

Construction Itérative

  • Retarder les décisions technologiques
  • Double boucle : BDD (externe) + TDD (interne)
  • Infra en dernier : ports puis adapteurs
  • ACL : protéger le domaine des systèmes legacy
Points clés :
  1. L'architecture hexagonale rend les tests naturels : chaque couche a son type de test
  2. Le domaine se teste sans aucun mock : preuve qu'il est bien isolé
  3. Le BDD avec Cucumber est le pont entre les métiers et les développeurs
  4. Les adapteurs in-memory permettent de retarder les décisions technologiques
  5. La double boucle BDD/TDD est le workflow idéal pour construire incrémentalement
  6. ArchUnit automatise la validation des règles architecturales
  7. L'ACL protège votre domaine des concepts des systèmes externes
