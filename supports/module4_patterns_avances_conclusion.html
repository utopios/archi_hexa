4.1 Event Sourcing - Principes fondamentaux

Dans une approche classique (CRUD), on stocke l'etat courant d'une entite. Avec l'Event Sourcing, on stocke tous les evenements qui ont conduit a cet etat. L'etat courant est recalcule en rejouant les evenements.

Approche CRUD

orders id=42 status=SHIPPED total=150EUR On ne sait pas QUAND ni COMMENT on est arrive la.

Approche Event Sourcing

event_store 1. OrderCreated {id=42, client="Dupont"} 2. ItemAdded {id=42, item="Laptop"} 3. OrderValidated {id=42} 4. PaymentReceived {id=42, amount=150EUR} 5. OrderShipped {id=42, tracking="XY123"} On sait TOUT ce qui s'est passe.
ProprieteDescription
ImmuabiliteLes evenements ne sont jamais modifies ni supprimes
Append-onlyOn ne fait qu'ajouter des evenements au journal
Source de veriteLe journal d'evenements EST la source de verite
RejouabiliteL'etat courant se reconstruit en rejouant tous les evenements
TemporaliteOn peut reconstruire l'etat a n'importe quel instant passe

4.1.2 Quand utiliser l'Event Sourcing

Cas d'usage favorables

  • Exigences d'audit : systemes financiers, bancaires, conformite reglementaire
  • Nombreuses transitions d'etat : gestion de commandes, workflows complexes
  • Reconstruire des etats passes : "Quel etait l'etat du portefeuille au 15 janvier ?"
  • Systemes distribues : se marie naturellement avec les architectures asynchrones
  • Detection d'anomalies : pouvoir rejouer et analyser

Quand NE PAS utiliser

  • Applications CRUD simples : formulaire de contact, catalogue statique
  • Pas de besoin d'historique : personne ne demandera "comment on est arrive la"
  • Peu de transitions d'etat : brouillon vers publie ne justifie pas ES
  • Equipes non formees : complexite accidentelle peut surpasser les benefices
  • Lecture temps reel : le rejeu peut etre lent (d'ou CQRS)
L'Event Sourcing ajoute de la complexite. Ne l'adoptez que si les benefices (audit, historique, rejouabilite) justifient le cout.

4.1.4 Implementation - DomainEvent & Evenements

Tout evenement du domaine implemente l'interface DomainEvent. Chaque evenement est un record Java (immuable par nature).

public interface DomainEvent { UUID eventId(); UUID aggregateId(); Instant occurredOn(); String eventType(); }
public record OrderCreated( UUID eventId, UUID aggregateId, Instant occurredOn, String clientName, String clientEmail ) implements DomainEvent { public OrderCreated(UUID orderId, String clientName, String clientEmail) { this(UUID.randomUUID(), orderId, Instant.now(), clientName, clientEmail); } @Override public String eventType() { return "OrderCreated"; } } public record ItemAdded( UUID eventId, UUID aggregateId, Instant occurredOn, String itemName, int quantity, BigDecimal unitPrice ) implements DomainEvent { public ItemAdded(UUID orderId, String itemName, int qty, BigDecimal price) { this(UUID.randomUUID(), orderId, Instant.now(), itemName, qty, price); } @Override public String eventType() { return "ItemAdded"; } }

4.1.4 Le port EventStore (port secondaire)

L'EventStore est un port secondaire dans l'architecture hexagonale. Le domaine definit le contrat, l'infrastructure fournit l'implementation.

package com.example.domain.port.out; /** * Port secondaire (driven) pour le stockage des evenements. * L'implementation concrete (PostgreSQL, EventStoreDB, Kafka...) * se trouve dans la couche infrastructure. */ public interface EventStore { /** * Persiste une liste d'evenements pour un agregat donne. * @param expectedVersion controle de concurrence optimiste */ void append(UUID aggregateId, List<DomainEvent> events, long expectedVersion); /** * Recupere tous les evenements d'un agregat (ordre chrono). */ List<DomainEvent> loadEvents(UUID aggregateId); /** * Recupere les evenements a partir d'une version (snapshots). */ List<DomainEvent> loadEventsFrom(UUID aggregateId, long fromVersion); }
Le port EventStore est defini dans le domaine. Les implementations (PostgreSQL, InMemory pour les tests, Kafka...) vivent dans l'infrastructure.

4.1.4 L'agregat qui se reconstruit

L'agregat Order maintient une liste d'evenements non persistes et expose des methodes qui emettent des evenements au lieu de modifier l'etat directement.

public class Order { private UUID orderId; private String clientName; private OrderStatus status; private List<OrderLine> lines; private BigDecimal totalAmount; private long version; private final List<DomainEvent> uncommittedEvents = new ArrayList<>(); // Factory : creer une nouvelle commande public static Order create(UUID orderId, String name, String email) { Order order = new Order(); order.applyNew(new OrderCreated(orderId, name, email)); return order; } // Factory : reconstruire depuis l'historique public static Order reconstitute(List<DomainEvent> history) { Order order = new Order(); for (DomainEvent event : history) { order.apply(event); order.version++; } return order; } // Methode metier public void addItem(String name, int qty, BigDecimal price) { if (status != OrderStatus.CREATED) throw new IllegalStateException("Impossible d'ajouter"); applyNew(new ItemAdded(orderId, name, qty, price)); } // Application d'evenement (rejeu) private void apply(DomainEvent event) { switch (event) { case OrderCreated e -> { this.orderId = e.aggregateId(); this.status = OrderStatus.CREATED; } case ItemAdded e -> { this.totalAmount = totalAmount .add(e.unitPrice().multiply(BigDecimal.valueOf(e.quantity()))); } case OrderShipped e -> { this.status = OrderStatus.SHIPPED; } default -> throw new IllegalArgumentException("Inconnu"); } } }

4.1.4 Service applicatif & lien hexagonal

Le service applicatif orchestre le chargement, l'execution de la logique metier et la persistance des evenements.

public class OrderCommandService { private final EventStore eventStore; public UUID createOrder(String clientName, String clientEmail) { UUID orderId = UUID.randomUUID(); Order order = Order.create(orderId, clientName, clientEmail); List<DomainEvent> newEvents = order.getUncommittedEvents(); eventStore.append(orderId, newEvents, 0); return orderId; } public void addItem(UUID orderId, String itemName, int quantity, BigDecimal unitPrice) { List<DomainEvent> history = eventStore.loadEvents(orderId); Order order = Order.reconstitute(history); order.addItem(itemName, quantity, unitPrice); eventStore.append(orderId, order.getUncommittedEvents(), order.getVersion()); } }
L'Event Sourcing s'integre parfaitement dans l'hexagonale : l'EventStore est un port secondaire, les evenements vivent dans le domaine, la publication (Kafka, RabbitMQ) est un adapteur driven.

4.2 CQRS - Principes

CQRS est un pattern architectural qui separe le modele d'ecriture (commandes) du modele de lecture (requetes).

Approche classique

Client -- Service -- Repository -- DB (read + write) (un seul modele)

Approche CQRS

CommandHandler -- WriteModel -- DB W Client QueryHandler -- ReadModel -- DB R Ecriture et lecture ont des modeles differents.
AspectCommand (Ecriture)Query (Lecture)
ButModifier l'etat du systemeRetourner des donnees
Retourvoid ou identifiant creeDTO / projection
ValidationRegles metier completesAucune (donnees deja validees)
ModeleRiche (agregats, entites, VO)Plat (optimise pour l'affichage)
FrequenceMoins frequentTres frequent (souvent 90% du trafic)

4.2.2 CQRS - Commandes et CommandHandler

Chaque commande est un record immuable. Chaque commande a exactement un handler.

// Interface marker pour les commandes public interface Command {} // Commande concrete public record PlaceOrderCommand( String clientName, String clientEmail, String itemName, int quantity, BigDecimal unitPrice ) implements Command {} // Handler generique public interface CommandHandler<C extends Command, R> { R handle(C command); } // Handler concret public class PlaceOrderCommandHandler implements CommandHandler<PlaceOrderCommand, UUID> { private final EventStore eventStore; @Override public UUID handle(PlaceOrderCommand cmd) { UUID orderId = UUID.randomUUID(); Order order = Order.create(orderId, cmd.clientName(), cmd.clientEmail()); order.addItem(cmd.itemName(), cmd.quantity(), cmd.unitPrice()); eventStore.append(orderId, order.getUncommittedEvents(), 0); return orderId; } }

4.2.2 CQRS - Requetes et QueryHandler

Le DTO de lecture est plat, optimise pour l'affichage. Pas d'agregat, pas de logique metier, juste des donnees.

// Interface marker pour les requetes public interface Query<R> {} // Requete concrete public record GetOrderSummaryQuery(UUID orderId) implements Query<OrderSummaryDTO> {} // DTO de lecture : plat, optimise pour l'affichage public record OrderSummaryDTO( UUID orderId, String clientName, String status, int itemCount, BigDecimal totalAmount, Instant lastUpdated ) {} // Handler de lecture public class GetOrderSummaryQueryHandler implements QueryHandler<GetOrderSummaryQuery, OrderSummaryDTO> { private final OrderReadRepository readRepository; @Override public OrderSummaryDTO handle(GetOrderSummaryQuery query) { return readRepository.findById(query.orderId()) .orElseThrow(() -> new OrderNotFoundException(query.orderId())); } }
Le modele de lecture n'a aucune logique metier. Il est optimise pour les requetes d'affichage et peut utiliser des technologies differentes (Elasticsearch, MongoDB, vues materialisees SQL).

4.2.2 CommandBus & QueryBus

Les bus routent chaque commande/requete vers son handler. En production, on utiliserait un framework (Axon, MediatR-like).

CommandBus

public class CommandBus { private final Map<Class<? extends Command>, CommandHandler<?, ?>> handlers = new HashMap<>(); public <C extends Command, R> void register( Class<C> type, CommandHandler<C, R> handler) { handlers.put(type, handler); } @SuppressWarnings("unchecked") public <C extends Command, R> R dispatch(C cmd) { CommandHandler<C, R> h = (CommandHandler<C, R>) handlers.get(cmd.getClass()); return h.handle(cmd); } }

QueryBus

public class QueryBus { private final Map<Class<? extends Query<?>>, QueryHandler<?, ?>> handlers = new HashMap<>(); public <Q extends Query<R>, R> void register( Class<Q> type, QueryHandler<Q, R> handler) { handlers.put(type, handler); } @SuppressWarnings("unchecked") public <Q extends Query<R>, R> R dispatch(Q q) { QueryHandler<Q, R> h = (QueryHandler<Q, R>) handlers.get(q.getClass()); return h.handle(q); } }

4.2.3 CQRS sans Event Sourcing

On peut utiliser CQRS sans Event Sourcing. Trois niveaux de separation :

Niveau 1 Meme base, modeles differents

Les commandes passent par l'agregat riche (JPA), les requetes par du SQL optimise (JdbcTemplate).

Niveau 2 Meme base, projections / vues materialisees

CREATE MATERIALIZED VIEW order_summaries AS SELECT o.id AS order_id, o.client_name, o.status, COUNT(l.id) AS item_count, COALESCE(SUM(l.quantity * l.unit_price), 0) AS total_amount FROM orders o LEFT JOIN order_lines l ON l.order_id = o.id GROUP BY o.id, o.client_name, o.status;

Niveau 3 Bases de donnees separees

Ecriture dans PostgreSQL (normalise), lecture dans Elasticsearch/MongoDB (denormalise). Synchronisation par Change Data Capture ou Domain Events.

4.2.4 CQRS avec Event Sourcing

La combinaison CQRS + ES est la plus puissante. Les evenements servent de modele d'ecriture, et des projections construisent les modeles de lecture.

Command --> CommandHandler --> Agregat --> EventStore (append-only) | evenements v Projection (Event Handler) | v Query <-- QueryHandler <-- ReadModel <-- Read Database (denormalise)
public class OrderSummaryProjection { private final OrderReadRepository readRepository; public void on(DomainEvent event) { switch (event) { case OrderCreated e -> onOrderCreated(e); case ItemAdded e -> onItemAdded(e); case OrderShipped e -> onOrderShipped(e); default -> { /* non gere par cette projection */ } } } private void onOrderCreated(OrderCreated event) { readRepository.save(new OrderSummaryDTO( event.aggregateId(), event.clientName(), "CREATED", 0, BigDecimal.ZERO, event.occurredOn())); } }
Coherence eventuelle (eventual consistency) : le modele de lecture peut avoir quelques millisecondes de retard sur le modele d'ecriture. Acceptable dans la plupart des cas.

4.3 Documentation vivante

4.3.1 Le probleme de la documentation classique

Semaine 1 : Documentation redigee --> Precise et a jour Semaine 4 : Quelques ecarts --> Globalement correcte Mois 3 : Documentation obsolete --> Trompeuse Mois 6 : Plus personne ne la lit --> Inutile Annee 1 : Document Word de 200 pages --> Dangereux (fausses certitudes)

Pourquoi elle echoue :

  • Elle est deconnectee du code : elle vit dans un wiki ou un document separe
  • Elle n'a pas de feedback : rien ne previent quand elle devient obsolete
  • Sa maintenance est manuelle et percue comme une corvee
  • Elle est souvent trop volumineuse pour etre lue
  • Personne n'a la responsabilite de la maintenir
Documentation vivante (Cyrille Martraire) : la meilleure documentation est celle qui ne peut pas devenir obsolete, parce qu'elle est generee ou verifiee automatiquement depuis le code.

4.3.2 Les tests comme documentation vivante

Les tests BDD avec Cucumber sont un excellent exemple de documentation vivante. Les scenarios Gherkin sont lisibles par les non-developpeurs et verifies a chaque build.

# order_management.feature # Ce fichier EST la documentation des regles metier. # S'il passe au CI, il est forcement a jour. Fonctionnalite: Gestion des commandes En tant que gestionnaire de commandes Je veux pouvoir gerer le cycle de vie des commandes Afin d'assurer un traitement fiable des achats Scenario: Creer une commande avec un article Etant donne un client "Jean Dupont" avec l'email "jean@example.com" Quand il cree une commande avec l'article "Laptop" a 1500.00EUR (quantite: 1) Alors la commande est creee avec le statut "CREATED" Et le montant total est de 1500.00EUR Scenario: Impossible d'expedier une commande non validee Etant donne une commande en statut "CREATED" Quand on tente de l'expedier Alors une erreur est levee avec le message "Impossible d'expedier une commande non validee"
Si le test passe au CI, la documentation est forcement a jour. Si le comportement change, le test casse et force la mise a jour.

4.3.2 ArchUnit - Tests d'architecture

ArchUnit permet d'ecrire des tests d'architecture. S'ils sont verts, l'architecture est conforme. S'ils cassent, quelqu'un a viole une regle.

class HexagonalArchitectureTest { private static JavaClasses classes; @BeforeAll static void setUp() { classes = new ClassFileImporter().importPackages("com.example"); } @Test @DisplayName("Le domaine ne doit dependre d'aucun framework") void domain_should_not_depend_on_frameworks() { noClasses().that().resideInAPackage("..domain..") .should().dependOnClassesThat() .resideInAnyPackage("org.springframework..", "javax.persistence..", "jakarta.persistence..") .check(classes); } @Test @DisplayName("Architecture hexagonale en couches respectee") void hexagonal_layers_are_respected() { layeredArchitecture().consideringAllDependencies() .layer("Domain").definedBy("..domain..") .layer("Application").definedBy("..application..") .layer("Infrastructure").definedBy("..infrastructure..") .whereLayer("Domain").mayOnlyBeAccessedByLayers( "Application", "Infrastructure") .whereLayer("Application").mayOnlyBeAccessedByLayers( "Infrastructure") .whereLayer("Infrastructure").mayNotBeAccessedByAnyLayer() .check(classes); } }

Un ADR (Architecture Decision Record) capture une decision architecturale : contexte, decision, consequences.

Les tests sont lents, le domaine est noye dans Spring. ## Decision Adopter l'architecture hexagonale pour le module de gestion de commandes. - Package domain : aucune dependance framework - Package application : use cases, depend uniquement du domaine - Package infrastructure : Spring, JPA, REST, implementations des ports ## Consequences + Le domaine est testable unitairement sans framework + L'infrastructure est interchangeable - Plus de code (interfaces port, mappers, DTOs) - Courbe d'apprentissage pour l'equipe

4.4.1 Le cout de la dette d'architecture

La dette d'architecture est bien plus dangereuse que la dette technique classique : invisible, couteuse, et tres difficile a rembourser.

SymptomeDescriptionImpact
Effet shotgunChaque changement touche de nombreux modulesRisque de regression
Tests lents et fragilesTests d'integration partoutCI lent, peur de refactorer
Onboarding longSemaines a comprendre le codeCout de recrutement
Peur du refactoring"On ne touche pas, ca marche"Code se degrade
Couplage frameworkMetier noye dans Spring/JPAMigration impossible
Big Ball of MudTout depend de toutArchitecture non maitrisee
Une equipe qui passe 40% de son temps a "travailler autour" d'une mauvaise architecture perd l'equivalent de 2 developpeurs sur une equipe de 5.

4.4.2 Scaler un SI avec l'hexagonale

Strategie 1 Scaling vertical

Optimiser l'interieur d'un hexagone sans changer l'architecture.

// Avant : adapteur simple public class JpaOrderRepositoryAdapter implements OrderRepository { public Optional<Order> findById(UUID id) { return jpa.findById(id) .map(OrderMapper::toDomain); } } // Apres : adapteur avec cache // Le port n'a pas change ! public class CachedOrderRepositoryAdapter implements OrderRepository { public Optional<Order> findById(UUID id) { Order cached = cache.get("order:" + id); if (cached != null) return Optional.of(cached); return jpa.findById(id) .map(OrderMapper::toDomain) .map(o -> { cache.set("order:" + id, o, Duration.ofMinutes(10)); return o; }); } }

Strategie 2 Scaling horizontal

Deployer plusieurs instances derriere un load balancer.

Grace au CQRS, on peut scaler independamment :

  • 1 instance en ecriture (commandes)
  • 10 instances en lecture (requetes) avec replicas

Strategie 3 Decomposition

Chaque bounded context est deja un hexagone isole. L'extraction en microservice est naturelle.

Monolithe hexagonal : [Orders] [Payments] [Notifications] dans un seul deploiement Microservices : [Orders Service] --> [Payments Service] --> [Notifications Service] (chacun est un hexagone avec sa propre DB)

4.4.2 Le Strangler Fig Pattern

Pour migrer progressivement d'un monolithe vers une architecture hexagonale, on utilise le Strangler Fig Pattern.

Phase 1 : Le monolithe gere tout [Monolithe : Orders + Payments + Notif] Phase 2 : On extrait un module (facade devant) [Gateway] --> Nouveau service Orders (hexagonal) --> Monolithe (Payments + Notif) Phase 3 : On continue l'extraction [Gateway] --> Service Orders (hexagonal) --> Service Payments (hexagonal) --> Monolithe (Notif seulement) Phase 4 : Le monolithe a disparu [Gateway] --> Service Orders --> Service Payments --> Service Notifications
Avantages du Strangler Fig : Migration progressive et sans risque (pas de big bang). On peut s'arreter a tout moment. Chaque nouveau module beneficie immediatement de la bonne architecture.

4.4.3 Quand NE PAS utiliser l'hexagonale

L'architecture hexagonale n'est pas une solution universelle.

SituationPourquoi pas l'hexagonaleAlternative
Application CRUD simpleL'overhead des ports/adapteurs ne se justifie pasArchitecture en couches classique
Prototype / MVPL'objectif est de valider une idee rapidementFramework RAD (Rails, Django)
Tres petite equipe (1-2 devs)Le couplage framework ne pose pas de probleme a cette echelleArchitecture simple, "clean enough"
Scripts et utilitairesUn script de migration n'a pas besoin de portsScript procedural classique
Domaine anemiqueSi la logique se resume a de la validation, l'hexagone sera videArchitecture en couches
Contraintes de delai extremesL'investissement initial ne sera pas amortiArchitecture la plus simple possible
Regle d'or : Commencez simple. Si le domaine se complexifie et que vous sentez la douleur du couplage, c'est le moment de refactorer vers l'hexagonale.
